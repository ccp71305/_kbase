## General Context for All Comments

This PR migrates the booking module from AWS SDK v1 (DynamoDBMapper) to AWS SDK v2 (Enhanced Client) via the `cloud-sdk-api` and `cloud-sdk-aws` shared libraries from `mercury-services-commons`. The same migration pattern has already been completed in **auth**, **network**, **registration**, **booking-bridge**, **webbl**, **self-service-reports**, **tx-tracking**, and **db-migration** modules.

The two most common categories of removed annotations are:

1. **`@DynamoDBDocument` (80 removals)**: In SDK v1, `@DynamoDBDocument` was required on every class that could be stored as a nested object in DynamoDB so that DynamoDBMapper knew how to serialize/deserialize its fields. In SDK v2 Enhanced Client, nested objects are handled by **custom converters** (`@DynamoDbConvertedBy`) which use Jackson `ObjectMapper` to serialize to native DynamoDB Map (M) format. The model classes themselves no longer need DynamoDB annotations — the converter controls the serialization.

2. **`@DynamoDBIgnore` (139 removals)**: In SDK v1, `@DynamoDBIgnore` was needed on utility/computed methods inside `@DynamoDBDocument`-annotated classes to prevent DynamoDBMapper from trying to persist them. Since `@DynamoDBDocument` is removed and converters use Jackson for serialization, only `@JsonIgnore` is needed (which was already present on these methods). The DynamoDB Enhanced Client does not scan these classes for getter methods — it only processes the top-level entity class (`@DynamoDbBean`).

## Comment 1: ConfirmPendingMessage.java

> **"How will the impact of the removed annotations be handled?"**

**Response**: `ConfirmPendingMessage` is a **pure DTO/POJO** — it is NOT a DynamoDB entity. It is used for inbound message deserialization from SQS/S3 JSON payload (booking confirmations from carriers in JPOA/EDIFACT canonical format).

The removed annotations were `@DynamoDBDocument` and `@DynamoDBIgnore`, which were inherited from the SDK v1 pattern where any class that appeared as a nested object inside a DynamoDB entity needed these annotations. In SDK v2, this class is never directly scanned by the Enhanced Client.

**How it's stored in DynamoDB**: `ConfirmPendingMessage` implements the `Contract` interface and is stored inside `BookingDetail.payload` via `ContractAttributeConverter`, which serializes the entire Contract tree to a compressed JSON string using Jackson `ObjectMapper`. The converter handles the `className:jsonData` format, class-level polymorphism via `Class.forName()`, and optional gzip compression for items >300KB.

**Jackson annotations are preserved** — all `@JsonProperty`, `@JsonIgnore`, and `@Valid` are untouched. No behavioral change.

**Verified by**: `ContractAttributeConverterTest` (unit tests for Contract serialization round-trip) and `BookingDetailDaoIT` (integration tests for full BookingDetail CRUD including payload).

## Comment 2: BookingDetail.java — Hash/Sort Key Annotations

> **"Why are the hashkey annotations removed? Is this handled in an alternative way? Referring to removed annotations @DynamoDBRangeKey and @DynamoDBAutoGeneratedKey"**

**Response**: The annotations are **replaced, not removed**. AWS SDK v1 annotations have direct SDK v2 equivalents:

| SDK v1 Annotation | SDK v2 Replacement | Location |
|---|---|---|
| `@DynamoDBHashKey` | `@DynamoDbPartitionKey` | `getBookingId()` |
| `@DynamoDBRangeKey` | `@DynamoDbSortKey` | `getSequenceNumber()` |
| `@DynamoDBAutoGeneratedKey` | Manual UUID generation in constructor | `calcBookingId()` → `UUID.randomUUID().toString()` |

The SDK v2 Enhanced Client does not support `@DynamoDBAutoGeneratedKey`. Instead, `BookingDetail` generates its UUID explicitly in the `calcBookingId()` factory method — functionally identical behavior. The old `getSortKey()` method has been refactored to `getSequenceNumber()` but produces the same formatted string pattern (`m_timestamp_state_ref`).

**Verified by**: `BookingDetailDaoIT` — All CRUD operations, GSI queries, and batch operations pass with the new annotation pattern. The `shouldSaveAndFindByBookingId` test confirms the UUID-based partition key works correctly.

## Comment 3: BookingDetail.java — "All the previously relevant annotations seem to be removed"

> **"All the previously relevant annotations seem to be removed? Is this handled in a different way?"**

**Response**: The annotations are **replaced with SDK v2 equivalents**, not simply removed. Here is the complete mapping:

| SDK v1 | SDK v2 | Applied To |
|---|---|---|
| `@DynamoDBTable` | `@Table(name = "BookingDetail")` + `@DynamoDbBean` | Class level |
| `@DynamoDBHashKey` | `@DynamoDbPartitionKey` | `getBookingId()` |
| `@DynamoDBRangeKey` | `@DynamoDbSortKey` | `getSequenceNumber()` |
| `@DynamoDBVersionAttribute` | `@DynamoDbVersionAttribute` | `getVersion()` |
| `@DynamoDBIgnore` | `@DynamoDbIgnore` | `getUniqueVersionId()`, `getIsSplitCopy()` |
| `@DynamoDBTypeConverted(...)` | `@DynamoDbConvertedBy(...)` | `getPayload()`, `getExpiresOn()`, `getAudit()`, `getEnrichedAttributes()`, `getMetaData()` |
| `@DynamoDBTypeConvertedEnum` | (Not needed — SDK v2 handles enums natively via `name()`) | `getState()`, `getChannel()`, `getPayloadType()` |
| `@DynamoDBAttribute` | (Not needed — SDK v2 auto-maps all bean properties) | All fields |

**Key point**: SDK v2 Enhanced Client uses `@DynamoDbBean` at the class level and then automatically maps all public getter/setter pairs to DynamoDB attributes. You only need explicit annotations for keys, converters, and fields to ignore — not for every field. This is a fundamental design difference from SDK v1's `DynamoDBMapper`.

**Verified by**: 27 tests in `BookingDynamoDbAdminCommandIT` (table creation with all GSIs), plus all `BookingDetailDaoIT` tests (full CRUD, GSI queries, batch operations).

## Comment 5: BookingDetail.java — @DynamoDBIgnore Removal

> **"Why are @DynamoDBIgnore annotations removed?"**

**Response**: `@DynamoDBIgnore` (SDK v1) is replaced by `@DynamoDbIgnore` (SDK v2) — different package but same purpose. The fields that need ignoring (`uniqueVersionId`, `isSplitCopy`) still have the SDK v2 annotation on their getters.

For the ~139 other `@DynamoDBIgnore` removals across nested model classes (DTOs inside `payload`, `enrichedAttributes`, etc.): these were only needed because SDK v1 required `@DynamoDBDocument` on nested classes and then DynamoDBMapper would try to persist every getter. Since these nested classes are now serialized via **converters using Jackson ObjectMapper**, DynamoDB Enhanced Client never scans them — it only processes the top-level `@DynamoDbBean` entity. `@JsonIgnore` (which was already present on these methods) controls what Jackson serializes.

## Comment 6: BookingDetail.java — DynamoHashAndSortKey Interface Removal

> **"Concerns on removal of `implements DynamoHashAndSortKey<String, String>`"**

**Response**: `DynamoHashAndSortKey<PK, SK>` was an interface from the old `dynamo-client` library (in the legacy `commons` dependency). It required two methods: `getHashKey()` and `getSortKey()` — which mapped to SDK v1's `@DynamoDBHashKey` and `@DynamoDBRangeKey`.

In the cloud-sdk migration, this interface is replaced by the entity annotation pattern:
- `@DynamoDbPartitionKey` on `getBookingId()` 
- `@DynamoDbSortKey` on `getSequenceNumber()`

The repository type `DatabaseRepository<BookingDetail, DefaultCompositeKey<String, String>>` from cloud-sdk-api defines the key structure. The old `getHashKey()`/`getSortKey()` methods are no longer needed because the Enhanced Client binds to the annotated getter methods directly.

This is the same pattern used by all other upgraded modules (network `Participant`, auth `TokenInfo`, registration `RegistrationDetail`, etc.).

## Comment 8: UnrecoverableAWSException.java

> **"Concern on the manner the exception is now handled and will it have any impact?"**

**Response**: `UnrecoverableAWSException` remains a `RuntimeException` with the same constructor signature `(String message, Exception e)`. The only change is that the Javadoc was updated to note it now wraps any `Exception` rather than specifically SDK v1's `SdkClientException`.

This is necessary because cloud-sdk clients throw their own exception types (wrapping SDK v2 `AwsServiceException`, `SdkClientException` from `software.amazon.awssdk.core.exception`), not SDK v1 exceptions. The calling code catches these and wraps them in `UnrecoverableAWSException`, preserving the same error propagation behavior.

**Callers are unchanged**: `SQSClient`, `SNSClient`, and `S3WorkspaceService` still throw `UnrecoverableAWSException` on failure, and all callers that catch this exception continue to work identically.

**Verified by**: `SQSClientTest` (11 tests), `SNSClientTest` (6 tests), `S3WorkspaceServiceTest` (15 tests) — all include error-path tests.

## Comment 9: EnrichedAttributes.java — @DynamoDBIgnore Removal

> **"Removal of @DynamoDBIgnore on certain methods have been flagged. Why are these removed? e.g. `getLocationAdditionalInfoForLocation()` — Are these now being persisted in the database?"**

**Response**: **No, these methods are NOT persisted.** The annotations are unnecessary because of the architectural change in how `EnrichedAttributes` is stored.

**Before (SDK v1)**: `EnrichedAttributes` had `@DynamoDBDocument`, so DynamoDBMapper scanned every getter and tried to persist it. Utility methods like `getLocationAdditionalInfoForLocation(Location)` needed `@DynamoDBIgnore` to prevent DynamoDBMapper from treating them as persisted attributes.

**After (SDK v2)**: `EnrichedAttributes` has **no DynamoDB annotations at all**. It is serialized via `EnrichedAttributesConverter` which uses Jackson `ObjectMapper.convertValue()` → `Map<String, Object>` → DynamoDB native Map. Only actual bean properties (with matching getters/setters and no `@JsonIgnore`) are serialized. The utility methods:
- `getLocationAdditionalInfoForLocation(Location)` — takes a parameter, not a bean property
- `getResolvedParty(PartyRole)` — takes a parameter, not a bean property
- `isMatch(...)` — private helper
- `compareEnumForNull(...)` — private helper

These are **not bean properties** (they require parameters), so Jackson and the Map serialization naturally exclude them. The existing `@JsonIgnore` annotations on these methods are an extra safety layer but technically redundant for parameterized methods.

**Verified by**: `EnrichedAttributesConverterTest` (unit) and `BookingDetailDaoIT.ConverterRoundTripTests.shouldRoundTripProductionRealisticEnrichedAttributes` (integration) — a production-realistic test with deeply nested structures (transactionPartyList, transactionLocationInfoList with Country/Geography, containerTypeList) all round-trip correctly through DynamoDB Local.

## Comment 10: BookingStateConverter.java — Dynamo Annotations Removed

> **"Why remove the dynamo annotations? Will the json be stored fine even if we remove this?"**

**Response**: `BookingStateConverter` **no longer participates in DynamoDB serialization**. It has been refactored from a `DynamoDBTypeConverter<BookingState, String>` (SDK v1) to a **Jackson `JsonDeserializer<BookingState>`** — its only role now is to handle legacy enum name mapping during JSON deserialization (e.g., `"BookingRequest"` → `BookingState.REQUEST`).

**How BookingState is stored in DynamoDB now**: The `BookingDetail.state` field (type `BookingState` enum) has no converter annotation. The SDK v2 Enhanced Client handles enums by default using `enum.name()` → String, which produces the same output as SDK v1's `@DynamoDBTypeConvertedEnum`.

**How BookingState is used in Contract payloads**: When deserializing `BookingDetail.payload` (a Contract JSON string), `ContractAttributeConverter` uses an `ObjectMapper` that has `BookingStateConverter` registered as a Jackson deserializer module. This handles backward compatibility for old enum names inside the JSON payload.

**Verified by**: `BookingDetailDaoIT` — all tests storing and retrieving BookingDetail with various states pass. `ContractAttributeConverterTest` — verifies Contract round-trip including BookingState fields.

## Comment 16: SpotRatesConverter.java — DynamoDBTypeConverter Removal

> **"The removal of `implements DynamoDBTypeConverter<String, SpotRates>` is flagged."**

**Response**: The converter has been replaced by `SpotRatesAttributeConverter` which implements SDK v2's `AttributeConverter<SpotRates>`:

| Aspect | Old: `SpotRatesConverter` | New: `SpotRatesAttributeConverter` |
|---|---|---|
| Interface | `DynamoDBTypeConverter<String, SpotRates>` (SDK v1) | `AttributeConverter<SpotRates>` (SDK v2) |
| Serialize | `convert(SpotRates) → String` | `transformFrom(SpotRates) → AttributeValue` |

The serialization logic is **identical** — JSON serialization with optional gzip compression for items approaching the DynamoDB 400KB limit. The old `SpotRatesConverter` class is retained (without the SDK v1 interface) as it may still be used as a utility in tests.

## Comment 17: CreateTables.java — Class Removal

> **"Create tables class for running create tables is being deleted. There could be upcoming scenarios where we need to create new tables for booking. I do not think deleting this is a good idea."**

**Response**: `CreateTables` is **replaced** by `BookingDynamoDbAdminCommand`, not simply deleted. The new command is **better** in every way:

### How the New Implementation Works

`BookingDynamoDbAdminCommand` extends `DynamoDbAdminCommand<BookingConfig>` from the `cloud-sdk-aws` library. It works as follows:

1. **Entity Registration**: A list of entity classes is passed to the parent constructor:
   ```java
   private static final List<Class<?>> ENTITY_CLASSES = List.of(
       BookingDetail.class, UniqueId.class, SequenceId.class,
       RapidReservation.class, TemplateSummary.class,
       SpotRatesDetail.class, SpotRatesToInttraRefDetail.class
   );
   ```

2. **Automatic Table Metadata Extraction**: The parent class uses the AWS SDK v2 Enhanced Client's `TableSchema.fromBean()` to extract:
   - Table name from `@Table(name = "...")`
   - Partition key from `@DynamoDbPartitionKey`
   - Sort key from `@DynamoDbSortKey` (if present)
   - GSIs from `@DynamoDbSecondaryPartitionKey` / `@DynamoDbSecondarySortKey` with `@GsiConfig`
   - TTL from `@TTL` annotation
   - Stream config from `@DynamoDBStream` annotation

3. **Table Creation**: Runs `dynamo-create` Dropwizard CLI command — creates all tables with GSIs, TTL, and streaming in one operation.

### How to Add New Tables in the Future

Simply:
1. Create a new entity class with appropriate annotations (`@DynamoDbBean`, `@DynamoDbPartitionKey`, etc.)
2. Add the class to `ENTITY_CLASSES` list in `BookingDynamoDbAdminCommand`
3. Run `java -jar booking-1.0.jar dynamo-create booking/conf/config.yaml`

This is **less error-prone** than the old approach where table definitions were hard-coded in Java with manual attribute definitions, GSI specifications, and throughput settings.

### Cross-Module Precedent

This exact pattern is already in production for:
- `AuthDynamoDbAdminCommand` (auth module)
- `NetworkDynamoDbAdminCommand` (network module)
- Registration, booking-bridge, webbl, tx-tracking, self-service-reports modules

### Old vs New Comparison

| Feature | Old `CreateTables` | New `BookingDynamoDbAdminCommand` |
|---|---|---|
| Table definition | Manual Java code specifying each attribute, key schema, GSI | Automatic extraction from entity annotations |
| GSI creation | Hard-coded attribute definitions and key schemas | Automatic from `@DynamoDbSecondaryPartitionKey` |
| TTL setup | Manual API call | Auto-detected from `@TTL` |
| Stream config | Manual | Auto-detected from `@DynamoDBStream` |
| Error handling | SDK v1 exceptions | cloud-sdk standardized error handling |
| Config | Direct SDK v1 client initialization | Via `DynamoDbClientConfig` from YAML |
| Idempotent | Needs manual check | Built-in to cloud-sdk |
| Adding new tables | Copy-paste table creation code | Add class to `ENTITY_CLASSES` list |

**Verified by**: `BookingDynamoDbAdminCommandIT` — 27 tests validating all 7 tables are created correctly with GSIs, TTL, streaming, and correct projection types. This is the most comprehensive table creation test suite in the project.
